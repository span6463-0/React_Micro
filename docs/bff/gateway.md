# BFF Gateway

The Backend for Frontend (BFF) is the sole entry point for the React frontend. It aggregates backend microservices, enforces authentication, and handles cross-cutting concerns.

**Port**: 4000  
**Base Path**: `/api`

---

## Architecture Role

```
React Apps  ──→  BFF Gateway (4000)  ──→  Auth API (5001)
                        │              ──→  User API (5002)
                        │              ──→  Item API (5003)
                        │
                  ┌─────┴────────────────┐
                  │ Cross-cutting Concerns│
                  │  · JWT Validation     │
                  │  · Rate Limiting      │
                  │  · Circuit Breakers   │
                  │  · Correlation IDs    │
                  │  · CORS              │
                  │  · Security Headers  │
                  └──────────────────────┘
```

---

## Middleware Stack

Requests are processed through the following middleware in order:

| Order | Middleware      | Purpose                                        |
| ----- | --------------- | ---------------------------------------------- |
| 1     | `correlationId` | Generates/propagates `X-Correlation-ID` header |
| 2     | `helmet`        | Sets security HTTP headers                     |
| 3     | `cors`          | Configures allowed origins                     |
| 4     | `express.json`  | Parses JSON request bodies                     |
| 5     | `cookie-parser` | Parses cookies (for refresh tokens)            |
| 6     | `rateLimiter`   | 100 req/15 min per IP on public routes         |
| 7     | `authenticate`  | Validates JWT on protected routes              |
| 8     | Route handlers  | Business logic delegation                      |
| 9     | `errorHandler`  | Centralized error formatting                   |

## Route Structure

```
/api/auth
  POST  /login          → authService.login()
  POST  /register       → authService.register()
  POST  /refresh        → authService.refresh()
  POST  /logout         → authService.logout()

/api/users              (requires authentication)
  GET   /               → userService.getAll()     [admin only]
  GET   /:id            → userService.getById()
  PUT   /:id            → userService.update()
  DELETE /:id           → userService.delete()     [admin only]

/api/items
  GET   /               → itemService.getAll()     [public]
  GET   /stats/overview → itemService.getStats()   [authenticated]
  GET   /:id            → itemService.getById()    [public]
  POST  /               → itemService.create()     [authenticated]
  PUT   /:id            → itemService.update()     [authenticated]
  DELETE /:id           → itemService.delete()     [authenticated]
```

## Authentication Flow

### Initial Login

```
1. Client POST /api/auth/login { email, password }
2. BFF forwards request to Auth API
3. Auth API returns { token, refreshToken, user }
4. BFF stores refreshToken in httpOnly cookie
5. BFF returns { token, user } to client
6. Client stores token in memory (Redux state)
```

### JWT Validation (per request)

```
1. Client sends request with Authorization: Bearer <token>
2. BFF extracts and verifies token against JWT_SECRET
3. If valid → attaches req.user = { id, email, role }
4. If expired → returns 401 (client triggers refresh flow)
```

### Token Refresh

```
1. Client detects 401 response
2. Client POST /api/auth/refresh (no body — cookie auto-sent)
3. BFF forwards refresh token to Auth API
4. Auth API rotates refresh token, returns new access token
5. BFF sets new httpOnly cookie, returns new access token
6. Client retries original request with new token
```

## Circuit Breaker Pattern

Each downstream service call is wrapped in an [Opossum](https://github.com/nodeshift/opossum) circuit breaker:

```javascript
// Example: Circuit breaker config
{
  timeout: 3000,        // Fail after 3s
  errorThresholdPercentage: 50,  // Open after 50% failures
  resetTimeout: 30000   // Try again after 30s
}
```

**States:**

- **Closed** — Normal operation, requests pass through
- **Open** — Service is failing; requests immediately return fallback error
- **Half-Open** — Test request sent; closes if successful, re-opens if it fails

## Request Correlation

Every request receives a `X-Correlation-ID` header (UUID v4). This ID is:

1. Generated by BFF (or forwarded if already present)
2. Attached to `req.correlationId`
3. Passed to all downstream API calls as `X-Correlation-ID`
4. Included in all log entries

This enables distributed tracing across services.

## Environment Variables

```env
# Server
NODE_ENV=development
BFF_PORT=4000

# JWT
JWT_SECRET=your-secret-key-min-32-chars
JWT_EXPIRES_IN=15m

# Downstream Services
AUTH_API_URL=http://localhost:5001
USER_API_URL=http://localhost:5002
ITEM_API_URL=http://localhost:5003

# CORS
CORS_ORIGINS=http://localhost:3000,http://localhost:3001

RATE_LIMIT_WINDOW_MS=900000
RATE_LIMIT_MAX=100
```
